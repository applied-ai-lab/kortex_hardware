kortex_hardware

Per-arm ros_control hardware interface for the Kinova Gen3, with Pinocchio gravity compensation and (new) soft-stop integration. Interactive documentation + vulnerability audit.

Overview

kortex_hardware is the per-arm hardware_interface::RobotHW implementation for the Kinova Gen3, written for the compliant control stack (the alternative to the upstream kortex_arm_driver). It opens a Kortex SDK session against one arm, exposes ROS-control joint handles, runs a 1100 Hz cyclic loop, and applies Pinocchio-based gravity compensation in effort/torque mode.

Two arms, two processes. The dual-arm Frank robot launches this binary twice — once each under /left_arm and /right_arm. Each process holds its own Kortex SDK session, its own TCP+UDP routers, and (now) its own soft-stop subscribers. All ros::Subscriber paths in the new soft-stop code use relative names so they resolve into the per-arm namespace.

main.cpp — line-by-line

src/main.cpp — 41 lines total

// Pre-soft-stop main.cpp was identical except for the write() gate at line 34.
#include <iostream>
#include <sstream>

#include <Gen3Robot.h>
#include <ros/rate.h>
#include "std_msgs/String.h"

int main(int argc, char* argv[])
{
  ROS_INFO_STREAM("Gen3 HARDWARE starting");
  ros::init(argc, argv, "kortex_hardware");
  ros::NodeHandle nh;

  Gen3Robot robot(nh);                                  // ① open Kortex sessions, register handles
  controller_manager::ControllerManager cm(&robot);  // ② plug robot into controller_manager
  bool use_admittance;
  ros::param::get("~use_admittance", use_admittance); // ③ ⚠ uninitialized if param absent — see Vulns

  ros::AsyncSpinner spinner(1);
  spinner.start();                                       // ④ start background spinner thread

  // Ros control rate of 1100Hz
  ros::Rate controlRate(1100);
  while (ros::ok())
  {
    robot.read();                                        // ⑤ pull last feedback from mLastFeedback into pos/vel/eff
    cm.update(robot.get_time(), robot.get_period());     // ⑥ tick all loaded controllers

    // Skip the cyclic write while the soft-stop is latched.
    // The firmware has already faulted the arm via ApplyEmergencyStop;
    // pushing further cyclic frames would just produce error spam.
    if (!use_admittance && !robot.estopLatched())            // ⑦ NEW: gate on the atomic latch
      robot.write();                                       // ⑧ push command into mBaseCommand → BaseCyclic::Refresh
    controlRate.sleep();
  }

  return 0;
}

Per-step rationale

① Gen3Robot robot(nh) — what happens inside
  • Reads private rosparams (~ip_address, ~username, ~password, timeouts).
  • Creates TCP and UDP transports (TransportClientTcp / TransportClientUdp) + their routers + their session managers.
  • Instantiates k_api::Base::BaseClient* mBase over the TCP router and k_api::BaseCyclic::BaseCyclicClient* mBaseCyclic over the UDP router. These are independent clients on independent sockets.
  • Allocates cmd_pos / cmd_vel / cmd_eff vectors, registers joint handles for arm + gripper finger with the appropriate JointStateInterface / EffortJointInterface / VelocityJointInterface / PositionJointInterface.
  • Registers a service server ~set_control_mode bound to setControlMode.
  • Calls CreateSession on both managers — throws std::runtime_error if the arm is unreachable.
  • Loads Pinocchio model from mURDFFile, precomputes gravity at the neutral pose.
  • Optionally sets CPU affinity and SCHED_FIFO priority on the main thread (compile-time gated on __has_include(<pthread.h>)).
  • NEW Attaches the three ros::Subscribers for in/emergency_stop, in/clear_faults, in/stop.
② controller_manager::ControllerManager cm(&robot)

Holds a non-owning pointer to the hardware interface. On each cm.update() it reads joint states from the registered handles, ticks the active controllers, and writes commands back into the cmd_pos / cmd_vel / cmd_eff arrays inside Gen3Robot. The controllers themselves are loaded dynamically via controller_manager's services — load_controller, switch_controller, etc.

③ ros::param::get("~use_admittance", ...)

If the param exists, populates use_admittance. If it doesn't, get returns false and use_admittance stays at its initial value — which is indeterminate because it's declared without an initializer. The if (!use_admittance ...) below reads UB. See vulnerability V-3.

④ ros::AsyncSpinner spinner(1); spinner.start()

Single background thread that services all ros::Subscriber callbacks, including the new soft-stop ones. The main thread does not spin — it just runs the 1100 Hz loop. This is the source of the threading complexity covered in the Threading model tab.

⑤ robot.read()

If this is the first cycle or admittance is on, calls mBaseCyclic->RefreshFeedback() to pull a fresh frame; otherwise reuses mLastFeedback. Translates Kortex's degrees/percent into SI radians and m, runs the input feedback through in_lpf (a low-pass filter), and writes into pos / vel / eff. The reason read() mostly doesn't call RefreshFeedback is that Refresh() (called by write()) returns a fresh feedback as a side effect, so reading from mLastFeedback is normally up-to-date.

⑥ cm.update(...)

Runs each active controller's update() at this iteration's timestamp. Controllers write desired commands into cmd_pos[] / cmd_vel[] / cmd_eff[]. Key: the controllers don't know whether the hardware interface will actually forward the commands — they always write, and write() decides whether to ship them.

⑦ Gate NEW

The conjunction !use_admittance && !robot.estopLatched() determines whether write() runs this cycle.

  • !use_admittance: pre-existing condition — admittance mode delegates write to another component, so the hardware iface skips its own write.
  • !robot.estopLatched(): NEW — when the soft-stop callback latched m_estop_latched=true, this cycle's write() is suppressed. Combined with the firmware-level ApplyEmergencyStop (which already faulted the arm), this prevents the cyclic loop from spam-pushing rejected frames at 1100 Hz.
⑧ robot.write() — what it actually does

Dispatches based on arm_mode (set via the ~set_control_mode service):

  • MODE_VELOCITY → sendVelocityCommand(cmd_vel) — calls mBase->SendJointSpeedsCommand (TCP RPC, NOT cyclic).
  • MODE_POSITION → sendPositionCommand(cmd_pos) — calls mBase->ExecuteAction (TCP RPC, blocking action — see vulnerability V-7).
  • MODE_EFFORT → sendTorqueCommand(cmd_eff) or sendCurrentCommand(cmd_eff) — applies gravity compensation, then calls mBaseCyclic->Refresh(mBaseCommand, 0) (UDP cyclic, true 1 kHz path).
  • EMERGENCY_STOP → sends zero torque/current via the same cyclic path.

The arm mode is sticky: last_arm_mode != arm_mode triggers a mode-switch dance involving ClearFaults, servoing-mode change between LOW_LEVEL_SERVOING and SINGLE_LEVEL_SERVOING, and per-actuator SetControlMode.

Destructor

Applies an emergency stop, clears faults, sets all actuators back to POSITION control, restores SINGLE_LEVEL_SERVOING, closes both sessions, deactivates routers, disconnects transports, and finally deletes the six router/session/transport pointers (but not mActuatorConfig, in_lpf, out_lpf, or finger — see V-4).

Soft-stop additions NEW

Three new ROS subscribers, three callbacks, one atomic latch, one gate in main.cpp. The full diff is small but the design has subtleties worth knowing.

Why the latch exists

The firmware's ApplyEmergencyStop faults the arm. With the arm faulted, BaseCyclic frames are rejected by the firmware — they don't cause motion, but they do return errors. At 1100 Hz that's 1100 errors per second of log noise. The atomic latch tells main.cpp to skip write() until ClearFaults runs, eliminating the spam.

Important: the latch is not the safety mechanism. The firmware's fault state is. If the latch is bypassed (e.g. logic bug), the arm still doesn't move — the firmware refuses the commands. The latch is bookkeeping.

The three callbacks

estopCallback — fail-safe path 
void Gen3Robot::estopCallback(const std_msgs::Empty::ConstPtr&)
{
  // Latch BEFORE issuing the RPC so even if it blocks briefly,
  // the next write() is already suppressed.
  m_estop_latched.store(true, std::memory_order_release);
  ROS_WARN("[kortex_hardware] EMERGENCY STOP received");
  try {
    mBase->ApplyEmergencyStop(0, {false, 0, 100});
  } catch (k_api::KDetailedException& ex) {
    ROS_ERROR("[kortex_hardware] ApplyEmergencyStop failed: %s", ex.what());
    // Latch stays TRUE — better latched-with-error than blindly resuming.
  }
}

Ordering matters: store before the RPC means a race where the spinner thread is preempted right after latching but before the firmware call still leaves the system in a safe state (no cyclic writes happening).

clearFaultsCallback — recovery path 
void Gen3Robot::clearFaultsCallback(const std_msgs::Empty::ConstPtr&)
{
  ROS_INFO("[kortex_hardware] CLEAR FAULTS received");
  try {
    mBase->ClearFaults();
    // Re-establish LOW_LEVEL_SERVOING.  Without this, fault clears
    // but the arm refuses subsequent cyclic frames silently.
    mServoingMode.set_servoing_mode(k_api::Base::ServoingMode::LOW_LEVEL_SERVOING);
    mBase->SetServoingMode(mServoingMode);
  } catch (k_api::KDetailedException& ex) {
    ROS_ERROR("[kortex_hardware] ClearFaults/SetServoingMode failed: %s", ex.what());
    return;  // keep latch on; safer to stay stopped than half-recovered
  }
  m_estop_latched.store(false, std::memory_order_release);
  ROS_INFO("[kortex_hardware] faults cleared, control loop re-enabled");
}

Two RPCs in sequence. Either failing keeps the latch on. The order — clear before servoing-mode — matches the recovery pattern at lines 644–660 and 838–866 of the existing code.

Gap: per-actuator SetControlMode (TORQUE/CURRENT) is not re-applied here. If the firmware fault demoted actuators to POSITION mode, the next cyclic frame in effort mode will have no effect. See V-6.

stopCallback — soft stop, no latch 
void Gen3Robot::stopCallback(const std_msgs::Empty::ConstPtr&)
{
  ROS_INFO("[kortex_hardware] SOFT STOP received");
  try { mBase->Stop(); }
  catch (k_api::KDetailedException& ex) { ROS_ERROR(...); }
}

Base::Stop is a non-faulting decel. The arm comes to rest but accepts cyclic frames again immediately — no clear-faults needed, no latching needed.

The Kortex BaseClient thread-safety question

The Kortex SDK ships its TCP traffic via an internal RouterClient that serializes outbound frames on a single queue. Concurrent RPCs from multiple threads are documented to interleave safely on this queue. The upstream kortex_arm_driver depends on this (its /in/* subscribers fire from the spinner thread while the main thread is mid-trajectory) and works in production. So our pattern of spinner-thread callbacks calling mBase->ApplyEmergencyStop() while the main thread is potentially mid-mBase->ExecuteAction() follows established convention.

That said, the SDK source is closed and the thread-safety claim is documentation, not a formal guarantee. If you want belt-and-braces, wrap mBase calls in a std::mutex m_base_mutex — adds ~5 lines, costs <1µs per acquire.

Detailed entries

Critical  V-1: destructor iterates with uninitialized mActuatorCount

File: Gen3Robot.h:178, Gen3Robot.cpp:414–417

// Gen3Robot.h
unsigned int mActuatorCount;                          // no in-class initializer

// Gen3Robot.cpp, destructor:
mControlModeMessage.set_control_mode(k_api::ActuatorConfig::ControlMode::POSITION);
for (int idx = 0; idx < mActuatorCount; idx++)    // ← UB if dtor runs before switchToEffortMode
  mActuatorConfig->SetControlMode(mControlModeMessage, idx + 1);

Path to bug: mActuatorCount is first assigned inside switchToEffortMode(). If the destructor runs before any effort-mode entry (e.g. Ctrl-C immediately after launch, or any ROS shutdown that fires before the first effort-mode trigger), the loop iterates with a garbage value — possibly billions of times, possibly accessing invalid actuator IDs and triggering Kortex SDK exceptions inside the dtor.

Fix: initialize mActuatorCount{0} at the declaration, OR query mBase->GetActuatorCount().count() in the ctor.

  throw ex_runtime;                                            // rethrow — but no cleanup
}

By the time the exception is rethrown, the ctor has already allocated m_tcp_transport, m_udp_transport, both routers, both session managers, mBase, mBaseCyclic, mActuatorConfig. None get freed before the exception propagates. The process exits shortly after, so memory is reclaimed — cosmetic issue, but if any of those types has a non-trivial destructor (open sockets, etc.) those side effects don't run.

Fix: same as V-4 — std::unique_ptr for the lot.

Med  V-6: clearFaultsCallback skips per-actuator mode restore

File: Gen3Robot.cpp:353–377

After ApplyEmergencyStop, actuators are demoted from TORQUE/CURRENT back to POSITION by the firmware. clearFaultsCallback re-applies SetServoingMode(LOW_LEVEL_SERVOING) but does not re-call mActuatorConfig->SetControlMode(TORQUE, idx) per-actuator. The existing recovery dance at lines 838–866 does both. Result: clear-faults appears to succeed, latch releases, write() resumes — but sendTorqueCommand sends torque commands to actuators in position mode → arm doesn't respond.

Fix: after SetServoingMode, iterate mActuatorCount and re-apply SetControlMode(TORQUE|CURRENT). Or factor lines 729–749 (switchToEffortMode) into a helper and call it.

High  V-7: sendPositionCommand uses blocking action at 1100 Hz

File: Gen3Robot.cpp:505–588

mBase->ExecuteAction(action);   // blocking action — not a streaming setpoint

ExecuteAction is a one-shot reach-joint-angles action that the SDK queues server-side and runs to completion. Calling it at 1100 Hz floods the TCP channel and the firmware action queue. Position mode under this hardware iface is effectively unusable as a streaming setpoint. (The early if (command == prev_cmd_pos) return; guard mitigates this only when commands don't change.)

Fix: use BaseCyclic position frames (mBaseCommand.mutable_actuators(i)->set_position(...) + mBaseCyclic->Refresh) for a true streaming position setpoint, the same pattern as torque/current modes.

Med  V-8: setControlMode has no return statement

File: Gen3Robot.cpp:447–476

bool Gen3Robot::setControlMode(...)
{
  if (req.mode == "position")      { arm_mode = ...; resp.success = true; }
  else if (req.mode == "velocity") { arm_mode = ...; resp.success = true; }
  // ... etc ...
  else {
    ROS_ERROR("Invalid control mode");
    resp.success = false;
  }
}                                                          // ← no `return` anywhere. UB caller value.

Declared as returning bool but no return on any path. Most compilers warn (-Wreturn-type); some at -O2 elide the warning and you read whatever's in EAX. ROS service callback's actual return semantics use resp.success, not the function return, so the practical impact is muted — but it's UB and breaks if anyone ever inspects the bool.

Additionally, arm_mode is written here from the spinner thread and read in write() from main without synchronization. Practically benign for an enum on x86 (read is atomic), but technically a data race.

Fix: return resp.success; at the end. Optional: make arm_mode a std::atomic<hardware_interface::JointCommandModes>.

Med  V-9: sendVelocityCommand empty-vector UB

File: Gen3Robot.cpp:602

for (size_t i = 0; i < command.size() - 1; ++i)   // wraps to SIZE_MAX if size == 0

If command is empty, size() - 1 underflows to SIZE_MAX and the loop iterates forever (or until command.at(i) throws). Empty command shouldn't normally occur — cmd_vel is sized to num_full_dof in the ctor — but if any code path zeroes the vector or default-constructs a temporary, this hangs the control loop.

Fix: if (command.empty()) return; guard at function entry.

High  V-10: Pinocchio URDF parse throws unhandled

File: Gen3Robot.cpp:270

pinocchio::urdf::buildModel(mURDFFile, model);   // throws on missing/bad file

If mURDFFile path is empty, unreadable, or malformed, buildModel throws. The ctor doesn't catch it; main doesn't either — the exception propagates out of main, terminating the process via std::terminate → SIGABRT. The crash log may not mention Pinocchio at all (depends on glibc backtrace verbosity).

Fix: wrap in try/catch; log a clear "URDF parse failed at <path>: <reason>" and re-throw.

Low  V-11: busy-wait in sendCurrentCommand

File: Gen3Robot.cpp:837–840

while (now - last < 1000) {
  now = GetTickUs();
}

Spins the main thread for up to ~1 ms with no std::this_thread::yield() or sleep. On a SCHED_FIFO RT thread on a dedicated core this is intentional (no risk of preemption). On a regular thread under load, this pegs the CPU. The comment above admits the design is a workaround for "ROS rate inaccuracy"; better fixed via nanosleep with a slack budget.

Low  V-12: controlRate.sleep() return value ignored

File: main.cpp:36

ros::Rate::sleep() returns false if the loop is over budget (the previous iteration took longer than the period). Ignoring this means the controller cadence silently degrades from 1100 Hz when the system is loaded — no warning, no metric. For a 1 kHz cyclic loop with hard real-time semantics, this is the kind of degradation you want to know about.

Med  V-13: race between soft-stop callback and ctor/dtor

File: main.cpp:15–21, Gen3Robot.cpp:322–329

The constructor attaches the subscribers near its end (line 322). But subscribers can fire as soon as they're attached — even before the constructor returns. If the spinner thread is already running (it isn't here — spinner.start() comes after Gen3Robot construction in main.cpp), or if the spinner starts before the ctor finishes its remaining work, an E-Stop callback could fire on a half-constructed object.

In current main.cpp ordering, the subscribers are attached before spinner start, so they queue silently until spinner.start(). That's safe today. But if anyone reorders or moves the subscriber attach earlier in the ctor, this assumption breaks.

More acute: in the destructor (~line 396), mBase is deleted at line 434. If a soft-stop subscriber callback fires during destruction, after the underlying ROS spinner has been torn down by global ros::shutdown but before the local Gen3Robot destructor completes, the callback could dereference dangling mBase. ros::AsyncSpinner's destructor joins its thread, so this is bounded — but the subscribers are member objects, destroyed before mBase, which means their internal callback queue is drained first. In practice safe; in principle a fragile ordering.

Fix: explicitly m_estop_sub.shutdown() at the top of the destructor before any mBase teardown. Eliminates the ordering subtlety.

Low  V-14: std::cout instead of ROS_* macros

Many error paths use std::cout or std::cerr (e.g. lines 517, 547, 624, 792, 879, 918). These bypass ROS log routing (rosout, file logging, log_level filtering) and can interleave between the spinner thread and main thread because std::cout isn't synchronized at byte-level by default. Garbled log lines under contention.

Fix: replace with ROS_ERROR / ROS_INFO. ROS's macros are thread-safe.

Med  V-15: first-feedback failure leaves mLastFeedback default-constructed

File: Gen3Robot.cpp:1011–1018

if (!mFirstFeedbackReceived || mUseAdmittance) {
  mLastFeedback = mBaseCyclic->RefreshFeedback();
  mFirstFeedbackReceived = true;                  // set even if Refresh throws... wait, no.
}

If RefreshFeedback() throws (network blip, firmware in fault), the assignment to mLastFeedback doesn't happen and mFirstFeedbackReceived is never set true. Subsequent calls to read() would re-attempt. Until one succeeds, mLastFeedback.actuators(i).position() reads from a default-constructed protobuf — usually a zero-size repeated field. Accessing actuators(0) on an empty repeated field is UB in protobuf-lite (debug builds assert, release builds return garbage).

Fix: guard read() on mFirstFeedbackReceived or on mLastFeedback.actuators_size() >= num_arm_dof; return early if not yet populated.

Soft-stop-specific risks (the new code)

Already-mitigated issues — for completeness
  • Latch-without-firmware-call: mitigated by latching before the RPC, so a preempted callback still leaves the system in fail-safe state.
  • Late subscriber: the soft-stop bridge publishes with latch=True, so a Gen3Robot starting after a stuck E-Stop receives the latched message on subscribe and immediately faults.
  • Concurrent ApplyEmergencyStop calls: idempotent at the firmware level — multiple calls land in the same fault state.
  • Spurious clear_faults: calling ClearFaults on a non-faulted arm is a documented no-op.
Residual risks — worth knowing
  • Per-actuator SetControlMode not re-applied (V-6) — silent failure mode.
  • mBase thread-safety is documentation-only — wrap in std::mutex if paranoid.
  • Resume jerk: controllers keep advancing their internal trajectory clock during the fault, then on resume the first cyclic frame may be far from current pose. Mitigated by the separate controller_estop_handler.py.

Recommended fix order

  1. V-1 (uninit mActuatorCount) — one-character fix, eliminates a real crash mode.
  2. V-3 (uninit use_admittance) — one-character fix, eliminates UB on the safety-critical gate.
  3. V-2 (silent partial init) — change return to throw so failures are loud.
  4. V-10 (URDF parse) — wrap in try/catch with a useful error message.
  5. V-15 (first-feedback) — guard read().
  6. V-6 (per-actuator mode restore) — copy the existing recovery dance.
  7. V-7 (position-mode action-spam) — replace with BaseCyclic position frames. Larger refactor.
  8. V-8 (no return + arm_mode race) — make atomic + add return.
  9. Everything else — cleanup, not crash mitigation.
